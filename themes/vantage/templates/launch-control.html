Phase 01 / Watch it fly

Launch Control

One app, downloaded once, can run thousands of different consoles. This is one of them: a live mission-control board for global spaceflight — real-time launch countdowns and a deliberately flaky API the interface rides straight through. And it's a folder of YAML, not a frontend you had to build.

Click the button and Vantage UI opens this example against our hosted server — no account, no setup, about thirty seconds. Hit "New simulated launch" and watch a rocket count down and fly while the board updates underneath you. The rest of this page is how that's put together, and why none of it took a line of UI code.

Phase 02 / It's just YAML

A view is a line, not a sprint

There is no frontend code in this example. The whole console is a handful of declarative files — one datasource, a table per entity, a page, an action — short enough to read in a sitting, and easy for you (or your coding agent) to change. The telemetry panel below isn't a component someone wrote; it's a list of small widgets, each a few lines of YAML.

Point at the server once. The api-client datasource learns the response envelope — the array lives under results, the total under count — and pushes filters server-side. One datasource drives all fifteen tables.

Model the relations once — the navigation falls out

Everything in this domain orbits one entity. A launch belongs to a provider, flies a rocket from a pad, carries payloads and crew, and ends in landings. Declare those relationships on the table and the entire navigation of the app — clickable cells, master/detail tabs — comes from them for free.

Phase 03 / Live, no plumbing

Real-time you declare, not wire up

Most builders make you reach for websockets, a subscription server and a state store to show data that moves. Here you set an interval and a cheap change probe, and the grid keeps itself current. The page is a binder — grid on top, auto-derived detail tabs below — with a toolbar action that triggers a launch. When the simulator moves a row, the board notices and refreshes; nothing else to run.

The "new launch" dialog is a table-derived form, so its foreign-key fields render as real dropdowns — and one is dependent: pick a provider and the rocket list narrows to that agency's configurations, from depends_on + filter_column alone.

Phase 04 / Flaky API, calm UI

A flaky server, a calm screen

Reliable data integration is where browser-based builders crack — a slow or failing API and the screen goes blank. So this server is built to misbehave: every read is delayed 150–1200ms and, about a tenth of the time, answered with a 503 instead of data, all while the simulator mutates rows underneath. The grid never blanks. The data path that survives this demo is the one that survives a real network.

dnsFlaky serverlatency + random 503s
cachedDiorama cacheholds last-good data; never blanks
grid_viewLaunch boardsmooth, virtualised, self-refreshing

The flakiness is honest middleware — random latency, then a probabilistic failure. It wraps only the read API; the trigger endpoint that starts a mission is left reliable, because writes must land.

warning Crank it up. Run the local server with serve --error-rate 0.3 to fail a third of all reads and watch the grid stay put. The non-blanking refresh is the whole reason this example exists.

Phase 05 / Driven by your API

Driven by your API — and you can run it all yourself

Vantage UI is mostly API-driven: it adapts to the API you already have. To prove it, the server here isn't a fixture — it's a real Rust backend on the same Vantage framework the UI consumes. Each entity collapses into one universal Vista facade — the exact read surface the UI's api-client speaks — and a single Axum handler serves the lot. "Consume an API" and "serve an API" turn out to be the same shape, mirrored.

Phase 06 / Extend it

Add a field in two lines

Low-code all the way down: a new piece of data is one expression on the server and one column in the YAML. Say you want each agency's success rate. It's an aggregate over the launches you already related — computed on read, never stored — so adding it is two small edits, no migration, no UI work.

On the server, declare the expression next to the counts that are already there. query_launches() is just the with_many relation reused as a subquery: model the relation once, aggregate over it however you like.

Phase 07 / Make it yours

The rocket is a stand-in for your launches

Everything you've seen — a generic app you download once, a console that's a folder of YAML, real-time data without plumbing, a UI that stays calm when the API doesn't, a field added in two lines — is the same shape you'd use to launch a campaign, ship a deploy, or run an overnight job. Less cinematic, identically built. And once it's a folder of YAML, you hand it around your organisation like any other file: check it in, diff it, distribute it.
